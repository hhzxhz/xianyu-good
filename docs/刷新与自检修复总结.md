# 刷新逻辑与自检修复总结

## 一、问题 1：刷新逻辑卡在主页「新发」而非「降价-新发」搜索页

### 1.1 原因分析

- **页面混淆**：原逻辑用「存在 新发 或 降价 任一」判断是否在搜索页。但**主页**顶部也有「新发」tab（关注、推荐、**新发**、省钱神券），仅凭「新发」会把主页误判为搜索页。
- **结果**：流程误入主页并点击「新发」后，一直停留在主页新发 tab，无法进入带「降价」「新发」的**搜索结果页**。

### 1.2 解决方案

| 项 | 说明 |
|----|------|
| **严格区分搜索页** | `is_on_new_drop_page()` 改为：**必须存在「降价」** 且存在「新发」或「综合」才视为搜索页，避免与主页新发混淆。 |
| **识别主页新发** | 新增 `is_on_home_new_tab()`：存在「新发」且无「降价」「综合」「区域」「筛选」，且存在「关注」「推荐」「省钱神券」等，则判定为主页新发。 |
| **恢复逻辑** | 在 `_ensure_on_search_list()` 中：若检测到 `is_on_home_new_tab()`，直接回首页并用 `ensure_home_then_search(keyword)` 重新进入搜索页，不再依赖多次 back。 |
| **自检与终止** | 在刷新循环中两处自检：① 每轮开始取商品前：若为主页新发或不在搜索页且恢复失败，则终止内层循环并打日志；② 每次 `refresh_search_list` 后：若为主页新发，同样终止内层循环。内层 break 后外轮询会重新执行「定位首页→搜索→新发」，实现自动重新定位。 |

### 1.3 涉及文件与方法

- **app/core/xianyu.py**
  - `_has_text_or_desc()`：统一按 text/description 判断控件是否存在。
  - `is_on_home_new_tab()`：是否在主页新发 tab。
  - `is_on_new_drop_page()`：是否在「降价-新发」搜索页（必须含「降价」）。
  - `_ensure_on_search_list()`：增加对主页新发的检测与回首页重搜。
- **app/core/buyer.py**
  - 新发页循环内：循环开始时的自检（主页新发 / 非搜索页且恢复失败则 break）。
  - 新发页循环内：`refresh_search_list` 后的自检（主页新发则 break）。

## 二、问题 2：刷新逻辑中程序经常「退出重来」

### 2.1 可能原因

- 抢购或返回列表时抛异常（如点击、返回、设备无响应），未捕获则导致整轮逻辑中断，外轮询重试，表现为「退出重来」。
- 刷新列表（`refresh_search_list`）或设备操作异常未捕获，同样会中断内层循环并触发外轮询重新定位。
- 解析商品列表已有多处 try/except 与超时，但抢购与返回路径此前未统一包裹，存在漏网异常。

### 2.2 解决方案

| 项 | 说明 |
|----|------|
| **抢购与返回包裹** | 对「点击商品并抢购 → 落库 → back_to_search_list」整段用 try/except 包裹；异常时打日志、尽量执行一次返回列表，然后 **break 内层循环**，由外轮询重新定位，避免未捕获异常导致线程级「退出」。 |
| **刷新列表异常** | 对 `refresh_search_list` 及紧随其后的自检用 try/except 包裹，异常时打日志，不向外抛，本轮回退后下一轮继续。 |
| **自检 break 行为** | 自检发现「主页新发」或「不在搜索页且恢复失败」时，统一用 break 结束内层循环，由外轮询重新执行 `ensure_home_then_search`（或 new_drop_only 下的重试），形成可预期的「重新定位」而非不可控退出。 |

### 2.3 涉及文件与方法

- **app/core/buyer.py**
  - 抢购流程：`click_item_at_index_and_buy`、落库、`back_to_search_list` 整体 try/except，异常时尝试返回列表并 break。
  - 刷新与自检：`refresh_search_list` + 自检已包在 try/except 中，仅打日志不抛异常。

---

## 三、自检与恢复流程简述

1. **每轮开始**：取商品前先 `set_current_keyword`，再自检：若 `is_on_home_new_tab()` → 打日志并 break；若 `not is_on_new_drop_page()` → 调用 `_ensure_on_search_list()`，仍失败则打日志并 break。
2. **刷新后**：执行 `refresh_search_list(current_keyword)` 后自检：若 `is_on_home_new_tab()` → 打日志并 break。
3. **break 后**：内层「新发页循环」结束，外轮询 `await asyncio.sleep(settings.poll_interval)` 后继续下一轮；下一轮会重新 `connect_device`、创建 driver，并执行 `ensure_home_then_search(task.keyword)`（from_app 模式），从而稳定回到「降价-新发」搜索页。

## 四、验证建议

- 从首页启动任务，观察是否稳定进入带「降价」「新发」的搜索页，且刷新多轮后仍保持在该页。
- 人为切到主页并点击「新发」tab，观察是否在下一轮或自检时被识别并自动重新定位到搜索页。
- 观察控制台是否出现「【自检】检测到处于主页「新发」tab」或「【自检】当前不在「降价-新发」搜索页且恢复失败」等日志，且之后任务能自动恢复。
- 观察抢购或返回列表异常时是否仅打「抢购或返回列表异常…将重新定位后继续」并重新定位，而不会出现未捕获异常导致任务线程退出。
